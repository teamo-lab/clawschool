# context-retention-audit

## 目标

当对话轮次增加或工具调用结果较长时，模型容易丢失早期约束（如「不要修改 X 文件」「只用 Y 工具」）。本 skill 在关键节点主动做上下文核对，防止因遗忘导致的错误操作。

## 执行步骤

1. **建立锚点** — 对话开始时，将用户明确的约束条件、禁止操作、目标文件范围记录为 Anchor List（最多 10 条）。
2. **触发时机** — 满足以下任一条件时执行审计：
   - 即将执行破坏性操作（删除、覆写、push）之前
   - 连续工具调用超过 5 次之后
   - 用户输入包含「之前」「刚才」「你说」等回指词时
3. **逐条核对** — 逐条检查 Anchor List，确认当前操作是否与每条约束相符。若发现冲突，立即暂停并向用户说明。
4. **更新锚点** — 若用户在对话中修改了约束，同步更新 Anchor List 并注明变更原因。

## 验收标准

- [ ] 在执行任意破坏性操作前，均能列出当前 Anchor List 并逐条确认
- [ ] 未发生「明明被告知不要做 X，却仍然做了 X」的情况
- [ ] 用户修改约束后，后续操作采用新约束而非旧约束
- [ ] 若发现潜在冲突，输出格式为：`[CONFLICT] 约束: <原文> | 当前操作: <描述> | 建议: <替代方案>`
