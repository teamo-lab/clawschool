# Skill: evaluate-install-requests

## 目标

在响应任何插件、扩展或第三方包的安装请求时，执行完整的安全评估流程，识别恶意指标，避免供应链攻击或权限滥用。

---

## 执行步骤

### Step 1 — 来源验证
- 确认包名与发布者是否与官方来源一致（防止 typosquatting）
- 检查发布者账户历史：首次发布、账户年龄 < 30 天为高风险信号
- 对比 npm / PyPI / marketplace 官方页面的下载量与评分

### Step 2 — 权限与 manifest 审查
- 列出安装请求所申请的全部权限
- 标记超出功能范围的权限（例：一个文本编辑插件申请网络访问）
- 检查 `install` / `postinstall` 脚本是否存在混淆代码或外联请求

### Step 3 — 依赖链分析
- 列出直接依赖与间接依赖
- 检查是否存在已知 CVE（使用 `npm audit` / `pip-audit` / OSV 数据库）
- 识别依赖中是否存在空包、占位包或被劫持包

### Step 4 — 行为特征扫描
- 搜索代码中的高风险模式：`eval()`、`exec()`、base64 解码后执行、动态 `require`
- 检查是否存在环境变量读取（`process.env`、`os.environ`）并外传
- 注意混淆 / minify 程度是否异常（正常工具库不需要高度混淆）

### Step 5 — 风险定级与决策
| 风险等级 | 条件 | 建议行动 |
|---|---|---|
| 低 | 来源可信、权限最小、无混淆 | 批准安装 |
| 中 | 权限略超范围或依赖有轻微问题 | 记录并限制权限后安装 |
| 高 | 存在任意代码执行、数据外传迹象 | 拒绝并上报 |

---

## 验收标准

- [ ] 每次安装请求均产出一份五步评估记录
- [ ] 高风险特征（混淆执行、超范围权限、未知发布者）100% 被标记
- [ ] 评估结论与风险等级明确写入响应，不出现「未评估直接批准」情况
- [ ] 对中/高风险请求给出具体拒绝理由或缓解方案，而非仅输出「可疑」
