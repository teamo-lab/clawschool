# structured-task-planning

## 目标

针对诊断中发现的复杂任务执行得分偏低问题，建立「先规划、再执行」的工作习惯，减少中途返工和遗漏步骤的情况。

## 触发条件

当收到满足以下任一条件的请求时，必须先输出执行计划再动手：
- 需要操作 3 个或以上文件
- 请求中包含「重构」「迁移」「批量」「全部」等关键词
- 需要调用 2 种或以上不同类型的工具

## 执行步骤

1. **理解意图** — 用一句话复述用户目标，确认理解无误。
2. **拆解子任务** — 将目标拆成有序的、原子化的子步骤，每步注明使用的工具或操作类型。
3. **标注风险点** — 对不可逆操作（删除、覆写、推送）加 `⚠️` 标记，并说明回滚方式。
4. **等待确认** — 输出计划后，明确询问「是否按此计划执行？」，不自动开始。
5. **按计划执行** — 获得确认后，严格按计划顺序执行，每完成一步用 `TodoWrite` 标记完成。
6. **偏差上报** — 如执行中发现计划需要调整，立即暂停并向用户说明原因，不静默修改计划。

## 验收标准

- [ ] 所有触发条件内的任务，在执行前都有可见的分步计划输出
- [ ] 含 `⚠️` 标记的步骤，执行前均获得用户明确确认
- [ ] 任务完成后，`TodoWrite` 中所有子任务状态均为已完成
- [ ] 无「静默跳过」或「静默合并」步骤的情况
