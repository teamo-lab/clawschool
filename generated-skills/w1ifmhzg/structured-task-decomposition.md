# structured-task-decomposition

## 目标

对于涉及 3 个以上独立动作的复杂请求，在执行前先输出带编号的执行计划，执行过程中逐步标记完成状态，避免遗漏步骤或乱序执行。

## 触发条件

用户请求中包含以下任一特征时启用：
- 包含「然后」「接着」「最后」等连接词
- 涉及多个工具（搜索 + 写文件 + 发通知 等）
- 任务描述超过 2 个独立目标

## 执行步骤

### Step 1 — 解析并拆解请求

阅读用户请求，识别所有独立动作，按依赖顺序排列：

```
并行可执行的步骤 → 标记为 [P]
必须串行的步骤   → 标记为 [S]
条件分支步骤     → 标记为 [C: 条件描述]
```

### Step 2 — 输出执行计划

在执行任何操作之前，先输出计划并等待隐式确认（继续执行即视为确认）：

```markdown
## 执行计划
1. [S] 读取配置文件，获取目标端点
2. [P] 并行调用 API-A 和 API-B
3. [S] 合并结果，写入报告文件
4. [C: 如有错误] 发送告警通知
```

### Step 3 — 逐步执行并更新状态

每完成一步，在输出中标记：

```
✓ Step 1 完成 — 端点: https://...
✓ Step 2 完成 — API-A: 200, API-B: 200
⏳ Step 3 进行中...
```

### Step 4 — 汇总结果

```markdown
## 执行摘要
- 总步骤: 4
- 完成: 4 / 失败: 0 / 跳过: 0
- 输出产物: report.md（已写入）
```

## 验收标准

- [ ] 执行前必须输出带编号的计划，不跳过此步
- [ ] 并行步骤使用并行工具调用，不串行等待
- [ ] 每步完成后有明确的状态标记（✓ / ✗ / ⏳）
- [ ] 最终输出执行摘要，包含失败数量
- [ ] 若某步失败，不静默跳过，必须说明原因并询问是否继续
