# api-response-validation

## 目标

针对 API 检查场景，建立规范的响应验证流程，确保每次 API 调用后对返回结果进行结构校验、状态码判断和异常处理，避免将错误数据透传给下游。

## 适用场景

- 调用外部 REST / GraphQL API 后需要验证结果
- 自动化任务中需要判断 API 是否返回有效数据
- 需要在异常响应时给出明确错误说明而非静默失败

## 执行步骤

### Step 1 — 检查 HTTP 状态码

```
- 2xx → 继续校验 body
- 4xx → 记录请求参数，提示调用方检查输入
- 5xx → 标记为服务端故障，不重试敏感操作，上报错误
- 超时 → 说明超时原因，询问是否重试
```

### Step 2 — 校验响应 Schema

对照预期字段列表逐项检查：

```
必填字段是否存在？
字段类型是否匹配（字符串/数字/布尔/数组）？
嵌套对象是否完整？
分页字段（total / page / next_cursor）是否一致？
```

如存在缺失或类型不符，输出差异列表，不继续处理。

### Step 3 — 业务语义校验

```
数值是否在合理范围内（如价格 > 0，时间戳为近期）？
枚举值是否在已知集合中？
关联 ID 是否能在上下文中对应到实体？
```

### Step 4 — 输出验证报告

按以下格式汇总：

```markdown
## API 验证报告
- 状态码: 200 ✓
- Schema 校验: 通过 / 失败（列出差异）
- 业务校验: 通过 / 警告（列出疑点）
- 结论: 数据可用 / 需人工复核 / 拒绝使用
```

## 验收标准

- [ ] 对 2xx / 4xx / 5xx 均有明确的不同处理路径
- [ ] 发现 schema 不符时不静默继续，必须输出差异
- [ ] 验证报告包含「结论」字段，非模糊表述
- [ ] 不对 5xx 响应执行破坏性重试（如 DELETE / PATCH）
