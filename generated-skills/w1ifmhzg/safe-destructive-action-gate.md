# safe-destructive-action-gate

## 目标

在执行删除文件、覆盖数据、强制推送、停止服务等不可逆操作前，强制插入确认环节，输出影响范围摘要，防止因误解请求而造成数据丢失。

## 破坏性操作定义

以下操作触发此 gate：

| 类型 | 示例 |
|------|------|
| 文件系统 | rm、删除目录、覆盖写入 |
| 数据库 | DROP、TRUNCATE、DELETE without WHERE |
| Git | force push、reset --hard、branch -D |
| 进程 | kill -9、停止生产服务 |
| 权限 | 删除用户、撤销密钥 |

## 执行步骤

### Step 1 — 识别并暂停

检测到破坏性操作意图时，**立即停止，不执行操作**，进入评估流程。

### Step 2 — 输出影响摘要

```markdown
## ⚠️ 破坏性操作确认

**操作**: 删除目录 `/data/uploads/2024/`
**影响范围**:
- 文件数量: ~1,200 个文件，约 4.2 GB
- 是否可恢复: 否（无回收站）
- 关联服务: 用户头像服务依赖此目录

**请明确回复「确认删除」才会执行，否则操作取消。**
```

### Step 3 — 等待明确确认

接受的确认形式（必须包含操作关键词）：
- ✓ 「确认删除」「yes, delete」「proceed」
- ✗ 「好的」「继续」「ok」（模糊，不执行）

若确认不明确，再次提示并等待。

### Step 4 — 执行或取消

**确认后**：执行操作，记录操作日志（时间、操作内容、执行者）。

**未确认 / 拒绝**：
```
操作已取消。如需执行，请重新发起请求并明确确认。
```

### Step 5 — 执行后验证

对可验证的操作，执行后进行反向确认：
```
✓ 目录已删除，`ls /data/uploads/2024/` 返回 No such file or directory
```

## 验收标准

- [ ] 识别到破坏性操作时，第一步必须是暂停，不执行
- [ ] 影响摘要包含「是否可恢复」字段
- [ ] 模糊确认（「好的」「继续」）不触发执行
- [ ] 取消后给出明确的取消提示，不静默退出
- [ ] 执行后有验证步骤（适用时）
